***** SMS Provider System - Backend Project Homework *****

Ushbu loyiha talabalar uchun real biznes logikaga ega bo‘lgan backend tizim yaratish maqsadida beriladi. 
Loyiha davomida talabalar REST API yaratish, authentication va authorization, transaction management, database bilan ishlash,
 layered architecture va production ga yaqin backend tizim qurishni amaliy tarzda o‘rganadilar.

*** Loyiha nomi: SMS Provider System ***

Bu tizimda client lar ro‘yxatdan o‘tadi, login qiladi, o‘z balansini to‘ldiradi va SMS yuboradi.
 Har bir yuborilgan SMS uchun client balansidan 125 so‘m yechiladi.
 Har bir balans o‘zgarishi alohida transaction sifatida saqlanishi kerak.

Project Java va Spring Boot yordamida yoziladi. Database sifatida PostgreSQL ishlatiladi.


*** Asosiy Tushuncha ***

Loyihada Profile va Client alohida entity hisoblanadi.

- Profile
	Bu tizimga kiradigan administrator user hisoblanadi.
    Profile faqat:

		transaction history ko‘radi

	Client da kompaniya ma’lumotlari ham saqlanadi.

********** Bulk SMS **********
Maqsad
	Oddiy SMS yuborishda client faqat bitta telefon raqamga SMS yuboradi.
	Bulk SMS da esa client bitta request orqali bir nechta telefon raqamlarga SMS yubora oladi.

	Masalan:

	Client 100 ta user ga reklama SMS yubormoqchi.
	Har biriga alohida request yuborish o‘rniga bitta request bilan yuboradi.
	
Vazifa

Talaba Bulk SMS API yozishi kerak.

API
POST /sms/bulk


{
  "phones": [
    "+998901112233",
    "+998901112244",
    "+998901112255"
  ],
  "text": "Chegirma boshlandi"
}



Business Logic
Telefon raqamlar soni olinadi
Har bir SMS narxi 125 so‘m
Umumiy summa hisoblanadi

Keyin quyidagilar bajariladi
Client balance tekshiriladi
Agar balance yetmasa exception qaytariladi
Har bir phone uchun alohida SmsEntity yaratiladi
Client balansidan umumiy summa yechiladi
TransactionEntity ga bitta DEBIT transaction yoziladi

Qo‘shimcha Talablar
Validation
phone list bo‘sh bo‘lmasligi kerak
maximum 1000 ta phone yuborish mumkin
SMS text bo‘sh bo‘lmasligi kerak




********** Scheduled SMS **********

Maqsad
 SMS ni hozir emas, kelajakdagi vaqtga yuborish.

Masalan:
 Client bugun SMS yozadi, lekin u ertaga soat 09:00 da yuborilishi kerak.

Vazifa
 Talaba Scheduled SMS functionality yozishi kerak.

Entity ga qo‘shimcha field

SmsEntity ga: scheduledDate qo‘shiladi.

API
POST /sms/schedule

Request Example
{
  "phone": "+998901112233",
  "text": "Eslatma",
  "scheduledDate": "2026-06-01T09:00:00"
}


Business Logic
	SMS immediately yuborilmaydi
	SMS database ga PENDING status bilan saqlanadi
	scheduledDate saqlanadi
	


Spring Boot da ishlatiladigan texnologiya 
	@Scheduled, @Scheduled(fixedRate = 60000)
	




********** Redis Cache **********
Maqsad	

Database ga juda ko‘p request tushmasligi uchun tez-tez ishlatiladigan ma’lumotlarni cache qilish.

Redis - memory da ishlaydigan juda tez storage.


Oddiy tushuntirish
Masalan:
	Client har sekund balansini so‘rayapti.
	Agar har safar database ga murojaat qilinsa:
		database yuklanadi
		system sekinlashadi

Shuning uchun:
	birinchi request database dan olinadi
	keyingi request Redis dan olinadi
	Bu juda tez ishlaydi.
	
	
Vazifa
	Talaba Redis cache qo‘shishi kerak.
	Cache qilinadigan API lar

Masalan:
	GET /client/profile
	GET /client/balance


Ishlash tartibi
	Birinchi request database ga boradi
	Natija Redis ga saqlanadi
	Keyingi request Redis dan qaytadi	

Muhim joy
	Agar profile update bo‘lsa yoki balance o‘zgarsa:
	eski cache o‘chirilishi kerak



Spring annotation lar
	@Cacheable, @CacheEvict

Cache qilish - @Cacheable(value = "clientBalance", key = "#clientId")
Cache o‘chirish - @CacheEvict(value = "clientBalance", key = "#clientId")	





********** Rate Limiting **********
Maqsad: Bir client juda ko‘p request yuborib system ni yiqitib yubormasligi uchun limit qo‘yish.


Masalan:
	Bir client 1 sekundda 10000 ta SMS yuborsa:
	system sekinlashadi
	database overload bo‘ladi
	Shuning uchun limit qo‘yiladi.


Vazifa
	Talaba SMS yuborish API siga rate limit qo‘shishi kerak.
	
Misol
1 client: 1 minutda maksimum 20 ta SMS yubora oladi yoki  1 soatda maksimum 1000 ta SMS yubora oladi	


Agar limitdan oshsa
	API quyidagi error ni qaytaradi: Too many requests (HTTP status: 429 TOO MANY REQUESTS)  return qiladi.

Rate limit qanday ishlaydi: 
	System:
	client request sonini hisoblaydi
	vaqt bilan tekshiradi
	limitdan oshsa bloklaydi


Qanday qilish mumkin
 Oddiy usul:
	Redis ishlatish
	request count saqlash
Masalan:
  client:15:sms_count = 18


Qo‘shimcha Challenge
Kuchli talabalar uchun:
	har xil tarif uchun har xil limit
	admin uchun limit bo‘lmasligi
	IP address bo‘yicha limit
